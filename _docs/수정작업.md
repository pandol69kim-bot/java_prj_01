## [2026-04-13] nginx `try_files` + `proxy_pass` 충돌 버그

### 파일
`nginx/nginx.conf`

### 문제
`location /` 블록에서 `proxy_pass`와 `try_files`를 함께 사용.  
`proxy_pass` 사용 시 `try_files`는 동작하지 않음. 외부 nginx에는 로컬 파일이 없으므로 항상 실패.

### 수정
```nginx
location / {
    proxy_pass http://frontend;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    # try_files 제거 — SPA 라우팅은 frontend 컨테이너 nginx.conf에서 처리
}
```

## [2026-04-14] 개발 컨테이너 무한 재시작 루프 — gradle bootRun + Spring DevTools 충돌

### 파일
- `docker-compose.dev.yml`

### 문제
`gradle bootRun` + `SPRING_DEVTOOLS_RESTART_ENABLED: "true"` 조합 시 무한 루프 발생:
1. Gradle이 `bootRun` 실행 → Spring Boot 기동 → .class 파일 생성
2. Spring DevTools가 클래스패스 변경 감지 → 앱 재시작
3. 재시작 시 Gradle이 재컴파일 → 새 .class 생성
4. DevTools가 다시 감지 → 무한 반복

### 수정
1. `SPRING_DEVTOOLS_RESTART_ENABLED: "false"` 설정
2. **CMD를 JAR 직접 실행으로 변경**:

```yaml
# 수정 전
command: gradle bootRun --no-daemon  (Dockerfile CMD)

# 수정 후 (docker-compose.dev.yml override)
command: >
  java -Djava.security.egd=file:/dev/./urandom
       -Dspring.profiles.active=dev
       -jar /app/build/libs/app-0.0.1-SNAPSHOT.jar
```

3. **JAR 빌드 방법**: `gradle bootRun` 대신 일회성 Gradle 컨테이너로 빌드

```bash
docker run --rm \
  -v "//d/AI바이브/...backend://app" \
  gradle:8-jdk17-alpine \
  sh -c "cd /app && gradle bootJar -x test --no-daemon -q"
```

빌드된 JAR: `backend/build/libs/app-0.0.1-SNAPSHOT.jar` (84MB)

### 근본 원인
개발 환경에서 `gradle bootRun`은 파일 감시·재컴파일 루프 문제가 발생하기 쉬움.
운영 JAR을 직접 실행하는 방식이 컨테이너 환경에서 더 안정적.

## [2026-04-14] `ddl-auto: create-drop` — 재시작 시 테이블 전체 DROP 버그

### 파일
`backend/src/main/resources/application-dev.yml`

### 문제
`spring.jpa.hibernate.ddl-auto: create-drop` 설정으로 인해 Spring Boot 종료 시  
모든 테이블이 DROP됨. Flyway 히스토리(`flyway_schema_history`)만 남고 나머지 테이블 소실.

### 원인
Flyway가 스키마를 관리하므로 Hibernate의 DDL 자동 관리는 불필요.  
`create-drop`은 테스트 환경 전용 설정으로 개발 환경에 잘못 적용됨.

### 수정
```yaml
# 수정 전
spring:
  jpa:
    hibernate:
      ddl-auto: create-drop

# 수정 후
spring:
  jpa:
    hibernate:
      ddl-auto: none
```

### 복구 절차
1. `flyway_schema_history` 레코드 삭제 → Flyway 재실행 유도
2. 마이그레이션 SQL 직접 실행 (V1~V5 테이블 생성)
3. `flyway_schema_history` 레코드 재삽입
4. 테스트 데이터 삽입

### 생성된 테이블 및 테스트 데이터

| 테이블 | 건수 | 내용 |
|--------|------|------|
| `users` | 1 | admin / admin1234 (ROLE_ADMIN) |
| `external_systems` | 3 | ORDER_SYS, STOCK_SYS (활성), LEGACY_SYS (비활성) |
| `external_data_syncs` | 16 | COMPLETED·FAILED·PENDING·PROCESSING·SKIPPED 각 상태 포함 |
| `sync_logs` | 14 | COMPLETED·PARTIAL_FAILED·FAILED, SCHEDULED·MANUAL, 최근 22시간 분산 |
| `refresh_tokens` | 0 | — |
