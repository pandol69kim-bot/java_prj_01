## 작업 내용

### 서브태스크 5.1: TaskScheduler 스레드풀 설정
- `SchedulerConfig`: `ThreadPoolTaskScheduler` Bean (poolSize=10, prefix=sync-scheduler-)
- 셧다운 시 실행 중 작업 최대 60초 대기
- 에러 핸들러로 스케줄러 예외 격리 (전체 스레드풀 영향 차단)
- `application.yml`에 `sync.scheduler.*` 설정 추가 (enabled, cron, maxRetry, batchSize)

### 서브태스크 5.2: DataSyncService
- `syncSystem()`: 단일 시스템 동기화 파이프라인 (fetch → 중복제거 → 배치저장 → SyncLog 기록)
- `syncAllActiveSystems()`: 활성 시스템 전체 순회 (스케줄러 호출용, 개별 예외 격리)
- `retryFailedRecords()`: FAILED 상태 + retryCount < maxRetry 레코드 재처리
- `SyncResult` record: total/success/fail 집계
- batchSize 단위로 나눠서 `saveAll()` 호출 (대용량 처리 최적화)

### 서브태스크 5.3: DataSyncScheduler
- `@ConditionalOnProperty(sync.scheduler.enabled)` — 환경별 활성화/비활성화
- `runFullSync()`: `@Scheduled(cron)` 매 10분 전체 동기화
- `runRetryFailed()`: `@Scheduled(cron)` 매 5분 실패 재처리
- 스케줄러 내부 예외는 catch하여 로그만 기록 (다음 실행에 영향 없음)

### 서브태스크 5.4: SyncMetricsService + SyncController
- `getSummary()`: 최근 24시간 SyncLog + ExternalDataSync 상태별 집계
- `SyncController`:
  - `GET /api/v1/sync/summary` — 현황 대시보드 데이터
  - `POST /api/v1/sync/trigger/{systemCode}` — 수동 즉시 동기화 (ADMIN)
  - `POST /api/v1/sync/retry-failed` — 실패 수동 재처리 (ADMIN)

- `@ConditionalOnProperty`로 dev/test 환경에서 스케줄러 자동 비활성화 가능
